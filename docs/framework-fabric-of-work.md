# Framework: Fabric of Work

## 1. Definition

The Autonomyx Framework is the **Fabric of Work**.

It defines how work is modeled, governed, routed, operated, evidenced, reconciled, delivered, and improved across people, agents, platforms, partners, customers, systems, and markets.

This is not only a software framework.

It is an operating framework for trusted commercial open source delivery at scale.

## 2. Core thesis

```text
Framework = Fabric of Work
Fabric = Meta-model of governed work
Platform = Orchestrator of work flow
Governance = Loop of trusted work
Model = Desired work state
Operator = Reconciler of work state
Policy = Constraint on work
Evidence = Proof of work
Registry = Source of truth for work
Marketplace = Published surface of trusted work
Runtime = Observed state of work
Drift = Work state mismatch
Reconciliation = Work made trustworthy again
```

## 3. What is work?

Work is any intentional state transition that creates, changes, validates, delivers, operates, or governs value.

Examples:

- A partner applies for certification
- A solution is submitted to the marketplace
- A connector requests trusted status
- A customer deployment is created
- A Juju model is updated
- A GitHub pull request changes policy
- A runtime incident is resolved
- A compliance requirement is satisfied
- A workflow is executed by an agent
- A human approves a high-risk action

Work is not only a task.

Work is a governed transition from one state to another.

## 4. Fabric of Work primitives

### Identity

Who or what performs, owns, approves, observes, or receives work.

Examples:

- Human
- Agent
- Partner
- Customer
- Organization
- Reviewer
- Operator
- System
- Marketplace publisher

### Intent

The reason work begins.

Examples:

- Submit partner profile
- Publish certified solution
- Deploy customer environment
- Resolve incident
- Change policy
- Upgrade runtime
- Renew certification

### Model

The desired state of the work.

Examples:

- Partner should be certified
- Solution should be listed
- Deployment should be compliant
- Runtime should be healthy
- Policy should be approved

### Policy

The constraints that shape whether work is allowed.

Examples:

- Required reviews
- Certification rules
- Branch protection
- Data handling rules
- Security baseline
- Licensing requirements
- Regional compliance controls

### Operator

The reconciler that moves work toward valid state.

Examples:

- GitHub runner
- Juju controller
- Kubernetes controller
- Human reviewer
- Partner delivery team
- Marketplace maintainer
- Compliance auditor
- Agent workflow

### Evidence

Proof that work happened correctly.

Examples:

- Status checks
- Review approvals
- Delivery evidence
- Audit logs
- Runtime status
- Incident reports
- Customer handoff
- Test results
- Certification review

### Registry

The source of truth for work state.

Examples:

- Partner registry
- Solution registry
- Connector registry
- Policy registry
- Certification registry
- Marketplace registry

### Runtime

Where work becomes operational reality.

Examples:

- Juju model
- Kubernetes cluster
- Customer environment
- Agent runtime
- Data platform
- Marketplace listing

### Drift

Difference between intended work state and observed work state.

Examples:

- Certified solution missing evidence
- Runtime unhealthy
- Partner metadata outdated
- Connector unsupported
- Policy changed without review
- Marketplace listing lacks support owner

### Reconciliation

The loop that brings work back into trusted state.

## 5. Flow of work

```text
Intent
  ↓
Model
  ↓
Policy
  ↓
Operator
  ↓
Evidence
  ↓
Governance
  ↓
Registry
  ↓
Runtime / Marketplace
  ↓
Observation
  ↓
Drift Detection
  ↓
Reconciliation
```

This is the Fabric of Work loop.

## 7. Juju in the Fabric of Work

Juju manages operational work.

```text
Controller = Operations control plane
Model = Desired operational boundary
Charm = Operational contract
Application = Managed capability
Unit = Workload instance
Relation = Dependency contract
Hook = Reconciliation event
Status = Observed operational state
```

Juju turns delivery work into continuous operations.

## 8. Platform in the Fabric of Work

The platform is the orchestrator of work flow.

It does not replace GitHub, Juju, Kubernetes, identity systems, marketplaces, or partner teams.

It coordinates them.

```text
Platform = Flow Orchestrator
```

The platform routes work across:

- identity
- policy
- agents
- humans
- CI/CD
- Juju
- registries
- marketplace
- observability
- compliance
- customer environments

## 9. Governance in the Fabric of Work

Governance is the loop that keeps work trusted.

```text
Governance = Observe + Compare + Constrain + Reconcile + Evidence + Approve + Publish + Monitor
```

Governance is not the final approval gate.

Governance begins when intent appears and continues after runtime publication.

## 11. Work at scale

At scale, work cannot depend on one central approval queue.

The Fabric of Work scales through:

- distributed operators
- shared fabric
- local governance loops
- global policies
- structured evidence
- federated partner delivery
- dynamic trust scoring
- automated drift detection
- human exception handling

## 12. Commercial open source meaning

Commercial open source is not open core.

In the Fabric of Work:

```text
Open source provides transparent work substrate.
Partners provide accountable work delivery.
Governance provides trust.
Evidence proves delivery.
Marketplace publishes capability.
Runtime proves operational reality.
```
